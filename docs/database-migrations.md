# 数据库迁移说明

服务启动时会自动执行元数据数据库迁移。迁移是幂等的，版本记录保存在 `schema_migrations` 表中。

## 支持的数据库

- SQLite：在单个事务中逐版本执行；适合开发和小规模内部部署。
- PostgreSQL：使用 advisory lock `FILEUPLD`，避免多实例同时启动时重复迁移；每个版本独立事务提交。

当前迁移版本：

| 版本 | 内容 |
|------|------|
| 1 | 创建当前基础表结构 |
| 2 | 为 `files` 增加软删除时间列及索引 |
| 3 | 增加审计日志查询索引 |

## 升级流程

1. 停止旧进程或准备滚动发布窗口。
2. 备份数据库、配置和文件数据；PostgreSQL 使用 `pg_dump`，SQLite 复制数据库文件前先停止服务。
3. 替换二进制并启动服务。
4. 查看启动日志，确认迁移完成；检查 `/health`、`/v1/admin/status` 和 Prometheus `fileupload_health_status`。
5. 通过 CLI 执行一次上传、下载和删除回归操作。

迁移失败时服务不会继续启动。不要手动删除 `schema_migrations` 或部分迁移列；先保留数据库副本并修复根因。

## 回滚

迁移只向前兼容，应用二进制回滚前必须确认旧版本能读取新 schema。生产回滚优先恢复到升级前的数据库备份和文件数据快照，而不是手工逆向删除列或索引。

SQLite 的单文件备份必须同时考虑 `-wal` 与 `-shm` 文件；最安全的方式是停止服务后复制数据库文件。PostgreSQL 回滚使用经过验证的 dump/备份恢复流程。
